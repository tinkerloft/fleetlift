# MCP Sidecar Design Spec

**Date:** 2026-03-15
**Status:** Approved
**Branch:** `mcp_sidecar`
**Scope:** Track E phases E1 + E2

---

## Problem

Agents in FleetLift sandboxes are black boxes. They can't access platform capabilities (context, knowledge, artifacts) during execution. All data is crammed into the prompt upfront; all outputs are collected post-execution. This limits agent effectiveness and prevents non-agentic tools from participating.

## Solution

A hybrid MCP shim model. A thin, stateless Go binary (`fleetlift-mcp`) is uploaded to each sandbox at provision time. It speaks MCP protocol locally (SSE/HTTP + stdio) and proxies every tool call to new FleetLift backend REST endpoints.

## Architecture

```
Sandbox
 ┌──────────┐  stdio or SSE  ┌──────────────┐
 │ Agent    │───────────────▶│ fleetlift-   │──────▶ FleetLift Backend
 │          │                │ mcp shim     │       POST /api/mcp/*
 └──────────┘                │ :8081        │
                             └──────────────┘
 ┌──────────┐  curl localhost      ▲
 │ Shell    │──────────────────────┘
 └──────────┘

Worker (ProvisionSandbox)
 1. Create sandbox
 2. Generate MCP JWT (team_id, run_id, aud: "mcp")
 3. Upload fleetlift-mcp binary via sandbox.WriteFile()
 4. Start shim: ./fleetlift-mcp --api-url $URL --token $TOKEN --port 8081
 5. Launch agent with MCP config
```

### Key Properties

- **Shim is stateless** — all business logic in the backend
- **Dual transport** — SSE/HTTP (default, started by ProvisionSandbox) + stdio (agent-launched via `--transport stdio`)
- **No image changes** — binary uploaded after sandbox provisioning
- **Graceful degradation** — if `FLEETLIFT_MCP_BINARY_PATH` unset, MCP skipped; agents run as today
- **Shell scripts** can skip the shim and call backend API directly with the same token

## Auth Model

### MCP JWT

Same signing key (`JWT_SECRET`), distinguished by audience claim:

```go
type MCPClaims struct {
    TeamID string `json:"team_id"`
    RunID  string `json:"run_id"`
    jwt.RegisteredClaims // aud: "mcp", exp: 24h
}
```

- Generated by `ProvisionSandbox` activity
- Long-lived expiry (24h) to survive sandbox TTL renewals
- Backend validates run is still active (not terminal) on every request — run lifecycle is the real access boundary
- Injected into sandbox as `FLEETLIFT_MCP_TOKEN` env var

### MCP Token Functions

`internal/auth/mcp.go` provides two dedicated functions (separate from existing `IssueToken`/`ValidateToken` which parse `Claims` with `UserID`/`TeamRoles`):

- `IssueMCPToken(secret, teamID, runID string) (string, error)` — signs JWT with `MCPClaims`, sets `aud: "mcp"`, 24h expiry
- `ValidateMCPToken(secret, tokenStr string) (*MCPClaims, error)` — parses into `MCPClaims`, rejects if `aud != "mcp"`

### MCPAuth Middleware

- Separate from existing user auth middleware (`internal/auth/mcp.go`)
- Uses `ValidateMCPToken` (not `ValidateToken`) to parse MCP-specific claims
- Checks run is not in terminal state (completed/failed/cancelled)
- Injects `team_id` and `run_id` into request context
- Handlers resolve `step_run_id` via `SELECT id FROM step_runs WHERE run_id = $1 AND status = 'running' LIMIT 1` — in fan-out, each sandbox has its own MCP shim so only one step is running per sandbox

## MCP Tools

### E1: Read-Only Context (3 tools)

| Tool | Endpoint | Input | Returns |
|------|----------|-------|---------|
| `context.get_run` | `GET /api/mcp/run` | *(from token)* | Run metadata, params, step graph with statuses |
| `context.get_step_output` | `GET /api/mcp/steps/{stepID}/output` | `step_id` | Output map + diff from completed upstream step |
| `context.get_knowledge` | `GET /api/mcp/knowledge?q=...&max=...` | `query`, `max_items` | Approved knowledge items for current workflow template |

### E2: Write Tools (4 tools)

| Tool | Endpoint | Input | Returns |
|------|----------|-------|---------|
| `artifact.create` | `POST /api/mcp/artifacts` | `name`, `content` (max 1 MB), `content_type` | `artifact_id` |
| `memory.add_learning` | `POST /api/mcp/knowledge` | `type`, `summary`, `details`, `confidence`, `tags` | `id`, `status` |
| `memory.search` | `GET /api/mcp/knowledge/search?q=...&tags=...` | `query`, `tags`, `max_items` | Knowledge items (team-scoped, cross-workflow) |
| `progress.update` | `POST /api/mcp/progress` | `percentage`, `message` | `ok` |

### Scoping

- All write tools derive `team_id` and `run_id` from the JWT — never sent by the client
- `step_run_id` resolved server-side (active running step for the run)
- `context.get_step_output` validates requested `step_id` belongs to the same `run_id`
- `context.get_knowledge` scoped to current workflow template — handler resolves `workflow_template_id` from the `runs` table using `run_id`
- `memory.search` scoped to team (cross-workflow discovery) — requires new `SearchByTeam(ctx, teamID, query string, tags []string, maxItems int)` method on `knowledge.Store` (SQL: `summary ILIKE '%' || $2 || '%' AND ($3::text[] IS NULL OR tags @> $3) AND status = 'approved'`)

### Not in scope (E3 — future)

- `inbox.request_input` — synchronous HITL bridge (MCP ↔ Temporal signals)
- `inbox.notify` — async notifications

## Shim Binary

**Package:** `cmd/mcp-sidecar/main.go`

~200 lines. No business logic.

```go
type Shim struct {
    apiURL     string
    token      string
    httpClient *http.Client
}

// Every tool handler follows this pattern
func (s *Shim) handleGetRun(ctx context.Context, args map[string]any) (any, error) {
    return s.call("GET", "/api/mcp/run", nil)
}

func (s *Shim) call(method, path string, body any) (map[string]any, error) {
    // Build request, set Authorization: Bearer token, send, parse JSON response
}
```

**CLI flags:**
- `--api-url` — FleetLift backend URL
- `--token` — MCP JWT
- `--port` — SSE/HTTP listen port (default 8081)
- `--transport` — `sse` (default) or `stdio`

**Build:** `go build -o fleetlift-mcp ./cmd/mcp-sidecar/` — static linux/amd64 binary. Separate Makefile target. Path configured via `FLEETLIFT_MCP_BINARY_PATH` env var on the worker.

## Backend Changes

### New Endpoints

All under `/api/mcp/*` with `MCPAuth` middleware:

```
r.Route("/api/mcp", func(r chi.Router) {
    r.Use(auth.MCPAuth(jwtSecret))

    r.Get("/run",                   mcp.HandleGetRun)
    r.Get("/steps/{stepID}/output", mcp.HandleGetStepOutput)
    r.Get("/knowledge",            mcp.HandleGetKnowledge)

    r.Post("/artifacts",           mcp.HandleCreateArtifact)
    r.Post("/knowledge",           mcp.HandleAddLearning)
    r.Get("/knowledge/search",     mcp.HandleSearchKnowledge)
    r.Post("/progress",           mcp.HandleUpdateProgress)
})
```

### Handler Package

`internal/server/handlers/mcp.go` — all 7 handlers in one file. Each handler:
1. Reads `team_id`, `run_id` from context (set by middleware)
2. Parses request params/body
3. Calls existing stores (DB, `knowledge.Store`)
4. Returns JSON response

### Error Response Format

All MCP endpoints return errors as `{"error": "message"}` with appropriate HTTP status codes (400 bad request, 401 unauthorized, 403 run terminated, 404 not found, 413 content too large, 500 internal). The shim maps non-2xx responses to MCP tool errors with the message.

### No New Database Tables

- Artifacts → existing `artifacts` table
- Knowledge → existing `knowledge_items` table
- Progress → writes to `step_runs.output` JSONB under `progress` sub-key via `jsonb_set(COALESCE(output, '{}'), '{progress}', ...)` (does not clobber agent's final output) + emits SSE event

## Provisioning Changes

`internal/activity/provision.go` gains 3 new steps after workspace setup:

1. **Generate MCP token** — `auth.IssueMCPToken(secret, teamID, runID)`, 24h expiry
2. **Upload shim** — read binary from `FLEETLIFT_MCP_BINARY_PATH`, upload via `sandbox.WriteBytes()` (new method — existing `WriteFile` takes `string`, which corrupts binary data), then `sandbox.Exec("chmod +x /usr/local/bin/fleetlift-mcp")`
3. **Start shim** — `sandbox.Exec("./fleetlift-mcp --api-url ... --token ... --port 8081 &")`
4. **Health check** — retry `sandbox.Exec("curl -sf http://localhost:8081/health")` up to 5 seconds before proceeding

Token and port injected as env vars: `FLEETLIFT_MCP_TOKEN`, `FLEETLIFT_MCP_PORT`.

**Graceful degradation:** If `FLEETLIFT_MCP_BINARY_PATH` not set or file doesn't exist, skip MCP setup. No breakage.

**`sandbox.Client` change:** Add `WriteBytes(ctx context.Context, id, path string, data []byte) error` to the interface. OpenSandbox implementation uploads via multipart form with binary content type.

### Agent Runner Changes

`ClaudeCodeRunner` adds MCP server config to Claude CLI invocation so Claude Code discovers the tools. Exact flag depends on Claude Code's SSE MCP client config format.

## Testing

### Shim Unit Tests (`cmd/mcp-sidecar/`)
- Mock HTTP server stands in for backend
- Each tool handler: correct endpoint, args marshalled, response returned, errors propagated
- Both transports start correctly

### Backend Handler Tests (`internal/server/handlers/mcp_test.go`)
- `httptest.NewRecorder()` with mock DB/stores
- MCP auth middleware: valid token, expired token, wrong audience, missing token, terminal run rejection
- Each handler: correct store calls, response format, cross-run access denied

### Provisioning Tests (`internal/activity/provision_test.go`)
- Memory sandbox client
- Verify upload, start, env var injection
- Verify graceful degradation when binary path unset

### Manual E2E
- Local stack via `scripts/integration/`
- Trigger workflow, verify agent can call MCP tools

## Not In Scope

- E3 interactive tools (`inbox.request_input`, `inbox.notify`)
- File-based artifact creation (reading from sandbox filesystem)
- Rate limiting on MCP endpoints
- External/user-defined MCP tools
- Drag-and-drop workflow builder UI
- Semantic search / embeddings for knowledge
